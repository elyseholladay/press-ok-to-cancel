WhatsApp vs iMessage • Feedback

Imagine we're building iMessage. We're like, hey, I have a great idea—let's add a progress bar so people know how fast their messages are being sent, and if they got through okay! But man, that blue progress bar can take foreevvverr—you know what I mean. It's not a bad piece of feedback to the user, but...

Whatsapp uses a green checkmark to indicate that a message has been sent and another to indicate that it’s been received. What the app does brilliantly is to show you the first checkmark immediately. It may not have been fully sent, but it will probably get there, so let's go ahead and show that check. This makes the user feel like Whatsapp is really fast at sending messages.

When I worked at Bazaarvoice, we worked on consumer product rating and review software. When a user submitted a review on a product, they could add related products, like a game to a review of a Wii. This feature had super-high conversion rates but was never used. I ran some user tests, and it turned out that the search was based on the number of times the search result came up—nothing else. So if you searched for “macbook” you would get “Macbook Case for Macbook Pro 13” and Macbook Pro 15”” rather than the actual Macbook, which might appear on page five.

So we built a demo and bumped search results based on the number of reviews a product has; most likely a Macbook Pro has more reviews than a case. Time to find the right product dropped from 2 minutes to :15-:45 seconds.

don't give users unoccupied time

speed

I read an article about the Houston airport, in which passengers were complaining that the wait to get their luggage was too long. They added more baggage people, got the bags off the planes and to the claim faster... and people still complained.

Turns out, it took passengers a minute to walk from their arrival gates to baggage claim and seven more minutes to get their bags. Roughly 88 percent of their time, in other words, was spent standing around waiting for their bags.

So the airport decided on a new approach: they moved the gates and baggage claim FARTHER away from each other, so it took six times longer to get there. Complaints dropped to near zero.

Occupied time (walking to baggage claim) feels shorter than unoccupied time (standing at the carousel).

and the worst kind of unoccupied time? waiting for the train schedule to load on your phone... while you are waiting for the train... while you are late to a work meeting... before you have had any coffee.

Critical Path

CSS blocks rendering; load them first
JS blocks downloads; load them last
The critical path is the path from the user following a link to the working experience. CSS blocks rendering, so you need to deal with it right away (i.e. at the top of the document, in your ). JS blocks downloads, so you need to deal with these last to ensure that they don’t hold up anything else on the page. It’s also worth noting that a browser will download all CSS before it will begin rendering your page --- http://csswizardry.com/2013/01/front-end-performance-for-web-designers-and-front-end-developers/ CSS is one of performance’s worst enemies, as outlined by Stoyan Stefanov, because of this rendering blockage. It’s also worth noting that a browser will download all CSS before it will begin rendering your page. This means that your print.css will be requested even if the browser is only rendering a page on screen. It means that any stylesheets that are only used based on a media query (e.g. ) will still be downloaded even if they aren’t needed.

Syotan Stefanov

scripts alter pages, so we have to wait for them

a browser will download as many assets as it can from a single domain in parallel. The more domains it is pulling from, the more assets can be downloaded, in parallel, at once. JavaScript interrupts this process, blocking parallel downloads from any and all domains, because: The script being called might alter the page, meaning the browser will have to deal with that before it can move on to anything else. In order for it to deal with that eventuality it stops downloading anything else in order to focus soleley on that. Scripts usually need to be loaded in a certain order for them to work, for example, loading jQuery before you load a plugin. Browsers block parallel downloads with JavaScript so that it doesn’t start downloading jQuery and your plugin at the same time; it should be pretty obvious that if you were to start downloading both in parallel, your plugin would arrive before jQuery would.
